LOGIC MODEL-BUILDING GAME – KONTEXT A PLÁN PRO CODEX
====================================================

ÚČEL DOKUMENTU
--------------
Tento soubor shrnuje dosavadní návrh projektu tak, aby na něj bylo možné přímo
navázat při implementaci ve VS Code s Codexem. Nejde o definitivní technickou
specifikaci. Projekt se má vyvíjet postupně a první verze má zůstat malá,
funkční a matematicky korektní.


1. OBECNÁ MYŠLENKA
------------------
Logic Model-Building Game má být interaktivní logická hra nebo výuková
platforma, v níž logika není pouze tématem, ale přímo herní mechanikou.

Hráč dostane formální podmínku a konstruuje matematický objekt, který ji
splňuje nebo vyvrací. Program řešení automaticky a deterministicky ověří.
Nemá tedy porovnávat hráčův výtvor s jedním předem uloženým „správným
obrázkem“. Různá korektní řešení mají být přijata.

Dlouhodobě by platforma mohla obsahovat různé režimy:

- Kripkeho modely pro modální logiku;
- epistemické modely;
- konečné algebraické modely;
- konečné modely logiky prvního řádu;
- tableau;
- konstrukci důkazů;
- převody mezi různými reprezentacemi.

První pilot ale bude pouze Kripke Model-Building Game pro základní modální
logiku. Ostatní režimy jsou jen výhledové a nemají komplikovat první verzi.


2. PRVNÍ PILOT: KRIPKE MODEL-BUILDING GAME
------------------------------------------
Hráč vizuálně vytváří konečný Kripkeho model

    M = (W, R, V),

kde:

- W je konečná množina světů;
- R je binární relace dostupnosti mezi světy;
- V přiřazuje atomickým výrokům světy, ve kterých platí.

Hráč bude moci:

- přidávat a odstraňovat světy;
- kreslit orientované relace dostupnosti;
- případně vytvářet reflexivní hrany;
- přiřazovat jednotlivým světům atomy p, q, r atd.;
- označit výchozí svět, ve kterém se formule vyhodnocuje;
- zadat nebo dostat modální formuli;
- zvolit nebo dostat cíl: vytvořit model, nebo kontramodel;
- spustit automatické ověření;
- získat srozumitelné vysvětlení výsledku.

Základní podporovaný jazyk:

    phi ::= p
          | not phi
          | phi and phi
          | phi or phi
          | phi -> phi
          | box phi
          | diamond phi

Rozhraní má ideálně přijímat běžné symboly:

    ¬  ∧  ∨  →  □  ◇

Praktické je později podporovat i textové alternativy jako !, &, |, ->,
box a diamond. Přesnou notaci lze upravit během vývoje.


3. DVA HLAVNÍ REŽIMY
--------------------

A. KAMPAŇ

Kampaň bude tvořena předem připravenými úrovněmi s postupně rostoucí
obtížností. Level neurčuje konkrétní správný model, ale formální zadání.

Příklad:

    Vytvoř kontramodel formule:
    □(p ∨ q) → (□p ∨ □q)

Validátor přijme libovolný model, ve kterém je zadaná formule v označeném
světě nepravdivá.

Možný postup kampaně:

1. atomické valuace v jednom světě;
2. první použití □ a ◇;
3. světy bez následníků a vakuózní pravdivost □phi;
4. větvení relace dostupnosti;
5. kombinace výrokových spojek a modalit;
6. vnořené modality;
7. hledání kontramodelů známých neplatných principů;
8. později případně reflexivní nebo tranzitivní rámce.

Level může mít krátkou nápovědu a vysvětlení po úspěšném řešení. Skóre lze
jednoduše odvozovat od počtu použitých světů a relací. Minimalitu řešení zatím
není potřeba dokazovat ani vyžadovat. Skóre má být spíše motivující metrika než
tvrzení o matematicky optimálním řešení.

B. SANDBOX

V sandboxu si uživatel sám:

- zadá formuli;
- zvolí, zda ji chce učinit pravdivou nebo nepravdivou;
- postaví vlastní Kripkeho model;
- označí hodnoticí svět;
- nechá řešení automaticky zkontrolovat.

Později může sandbox obsahovat i režim, ve kterém uživatel pouze vytvoří model
a sleduje pravdivost formule ve všech světech.


4. AUTOMATICKÁ VERIFIKACE
-------------------------
Verifikace nebude používat GPT jako rozhodčího. Pro konečný model stačí vlastní
malý deterministický algoritmus implementující standardní Kripkeho sémantiku.

Základní případy:

- atom p je pravdivý ve světě w právě tehdy, když je p součástí valuace w;
- negace obrací pravdivostní hodnotu;
- spojky se vyhodnocují standardně;
- □phi platí ve w právě tehdy, když phi platí ve všech světech dostupných z w;
- ◇phi platí ve w právě tehdy, když phi platí alespoň v jednom světě
  dostupném z w.

Formule se nejprve tokenizerem a parserem převede na abstraktní syntaktický
strom (AST). Evaluator potom strom rekurzivně vyhodnotí v konkrétním světě.

Validátor kampaně zkontroluje:

- zda je formule syntakticky správná;
- zda existuje označený výchozí svět;
- zda je vytvořený model interně korektní;
- zda je formule v daném světě pravdivá nebo nepravdivá podle cíle levelu;
- případně počet světů a relací pro skóre.

Důležitá budoucí funkce je vysvětlení výsledku. Například:

    □p je ve w0 nepravdivé, protože w0 R w2 a p ve w2 neplatí.

To lze vytvořit přímo z evaluatoru bez AI: evaluator nemusí vracet jen
true/false, ale také strukturované informace o rozhodující podformuli a světě.


5. DOPORUČENÁ TECHNICKÁ PODOBA
------------------------------
Projekt má být webová aplikace běžící primárně v prohlížeči.

Doporučený základ:

- React pro uživatelské rozhraní;
- TypeScript pro implementaci a typovou kontrolu;
- Vite pro vytvoření a spouštění projektu;
- React Flow nebo podobná knihovna pro vizuální editor světů a hran;
- Vitest pro automatické testy;
- localStorage pro lokální ukládání postupu kampaní.

Pro první pilot není potřeba:

- Python;
- vlastní backend;
- databáze;
- OpenAI API;
- externí theorem prover;
- SAT/SMT solver;
- placený hosting.

Pythonový prototyp by byl možný, ale není příliš účelný, protože cílová
aplikace má být interaktivní web a jednoduchý evaluator lze přímo napsat v
TypeScriptu. Python nebo externí solver mohou být relevantní až pro budoucí
složitější automatické hledání modelů, generování úloh nebo jiné sémantiky.

Uživatel projektu zatím nezná React, TypeScript ani Vite. Codex má proto:

- volit jednoduchou a čitelnou architekturu;
- nepřidávat zbytečné frameworky a abstrakce;
- vysvětlovat důležité příkazy a strukturu projektu;
- udržovat kvalitní README;
- doplňovat testy ke každé části logického jádra;
- po změnách projekt sám spouštět, testovat a opravovat zjištěné chyby.


6. DOPORUČENÉ POŘADÍ IMPLEMENTACE
---------------------------------

ETAPA 1 – LOGICKÉ JÁDRO

Testy mají zahrnout minimálně:

10. JINÉ BUDOUCÍ LOGICKÉ REŽIMY
-------------------------------
Stejný obecný herní princip lze později použít i mimo Kripkeho sémantiku:

A. Algebraic Semantics Builder

Hráč konstruuje malou konečnou algebru nebo uspořádanou strukturu, doplňuje
operace a hledá valuaci, při které zadaná formule nebo identita selže. Mohlo by
jít například o Booleanovy algebry, Heytingovy algebry, residuované svazy nebo
orthomodulární svazy. Validátor musí nejprve ověřit, že struktura skutečně
splňuje axiomy požadovaného typu algebry.

B. Finite First-Order Model Builder

Hráč volí konečnou doménu a interpretuje konstanty, relace a funkce tak, aby
splnil nebo vyvrátil zadanou množinu formulí.

C. Tableau Game

Hráč správně rozvíjí větve tableau. Cílem může být uzavřít všechny větve nebo
z otevřené větve extrahovat model.

D. Proof Construction Game

Hráč konstruuje důkaz v přirozené dedukci, sekventovém kalkulu nebo jiném
formálním systému a program kontroluje každý krok.

E. Translation Challenges

Úlohy propojující formuli, Kripkeho model, algebraickou reprezentaci, tableau
a důkaz. Tyto režimy mohou být odborně velmi zajímavé, ale nemají být
implementovány současně s prvním modálním pilotem.


11. ZÁSADY PRO DALŠÍ PRÁCI
--------------------------

- Matematickou správnost kontroluje deterministický kód, nikoliv GPT.
- Nejdříve vznikne malé funkční jádro, potom vizuální a herní vrstvy.
- Kampaň nesmí vyžadovat pouze jedno předem uložené řešení.
- Minimalitu modelů v první verzi neřešíme.
- Skóre podle počtu světů a relací je pouze herní metrika.
- Projekt nemá předčasně zavádět backend, AI API ani solver.
- Architektura má umožnit pozdější rozšíření, ale nemá kvůli nim nafukovat MVP.
- Každá důležitá část logického jádra musí mít automatické testy.
- Codex má po každé větší změně spustit relevantní testy a build.
- Dokument se může během implementace aktualizovat podle skutečně přijatých
  rozhodnutí.


12. DOPORUČENÝ PRVNÍ ÚKOL PRO CODEX VE VS CODE
----------------------------------------------
Po vytvoření prázdného repozitáře lze začít přibližně tímto zadáním:

    Read logic_model_building_game_kontext.txt and treat it as the current
    project context. Start only with Stage 1: initialize a minimal React +
    TypeScript + Vite application and implement the formal logic core without
    a graphical editor.

    Define a typed AST for the stated modal language, a clear representation
    of finite Kripke models, and a deterministic recursive evaluator for
    M,w |= phi. Add comprehensive Vitest unit tests, including terminal worlds,
    vacuous truth of box, diamond failure, branching frames, nested modalities,
    cycles and reflexive edges.

    Keep the architecture simple and readable. Do not add a backend, OpenAI
    API, external solver, campaign system or graphical model editor yet.
    Document the mathematical conventions and the commands needed to install,
    test and run the project in README.md. Run the tests and build before
    reporting completion.
